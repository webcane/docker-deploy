    **1. Create .golangci.yml at repo root (per D-01, D-02, D-03):**

    Create a .golangci.yml file with these exact contents:

    ```yaml
    linters:
      disable-all: true
      enable:
        - errcheck
        - govet
        - staticcheck
        - goimports

    linters-settings:
      goimports:
        local-prefixes: github.com/webcane/docker-deploy

    issues:
      max-issues-per-linter: 0
      max-same-issues: 0
    ```

    Rationale for each linter per D-02:
    - errcheck: catches unhandled errors (critical for a CLI tool doing SSH/SFTP)
    - govet: catches common mistakes (misuse of sync.Mutex, printf format mismatches)
    - staticcheck: broader static analysis (deprecated APIs, bugs, simplifications)
    - goimports: enforces import grouping with project imports in a third group per D-03

    `disable-all: true` ensures only the four specified linters run — no surprise linters added by golangci-lint version bumps.

    `local-prefixes: github.com/webcane/docker-deploy` groups project imports as a third import group, separate from stdlib and third-party.

    **2. Add make lint and make fmt to Makefile (per D-04):**

    Update the .PHONY line to include lint and fmt:
    ```
    .PHONY: build install test test-ci lint fmt
    ```

    Add after the existing `test-ci:` target:
    ```makefile
    lint:
    	golangci-lint run ./...

    fmt:
    	goimports -w -local github.com/webcane/docker-deploy ./...
    ```

    Use a hard tab for indentation (Makefile requires tabs). The `make lint` target runs golangci-lint directly — golangci-lint reads .golangci.yml from the repo root automatically. The `make fmt` target runs goimports with the same local prefix as configured in .golangci.yml for consistency.

    Preserve all existing Makefile targets unchanged.

  <action>
    Add a `lint` job to ci.yml per D-04. The lint job runs in parallel with the existing `test` job (no `needs:` dependency on test). The integration job's `needs: [test]` remains unchanged — integration waits for test, but does NOT need to wait for lint.

    Add the following job to ci.yml (insert before or after the `test` job — parallel position):

    ```yaml
      lint:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - uses: actions/setup-go@v5
            with:
              go-version-file: go.mod

          - name: Install golangci-lint
            run: go install github.com/golangci/golangci-lint/cmd/golangci-lint@v1.64.8

          - name: Run lint
            run: make lint
    ```

    Pin golangci-lint to `@v1.64.8` (current stable v1.x at time of planning, 2026-05) rather than `@latest`. Using `@latest` in CI is non-reproducible — a new golangci-lint release can add default-enabled linters or change existing linter behavior, causing CI failures on unchanged code. The pinned version ensures the lint gate is stable between CI runs. Update the version when Dependabot or a manual bump is warranted.

    The lint job does NOT need the `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24` env var (that is being removed by 11-02-PLAN.md anyway, and this plan is Wave 1 alongside 11-02).

    After adding the lint job, ci.yml will have three jobs: test, lint (both Wave 1, parallel), and integration (blocked on test).

    Do NOT add `needs: [lint]` to the integration job — lint failures block PRs via branch protection, not by blocking integration from running.
