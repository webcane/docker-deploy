    PREREQUISITES.md content:

    H1: "# Prerequisites"
    Intro: "docker-deploy connects to your VPS over SSH. You need an SSH key pair and a remote user configured with appropriate permissions."

    H2: "## 1. SSH Key Setup"
    Step-by-step:
    1. Check if you already have a key: ls ~/.ssh/id_ed25519.pub (or id_rsa.pub)
    2. Generate if missing: ssh-keygen -t ed25519 -C "your-email@example.com" — accept default path, set passphrase (recommended)
    3. Copy to VPS: ssh-copy-id sshuser@vps.example.com OR manually append ~/.ssh/id_ed25519.pub to ~/.ssh/authorized_keys on the remote
    4. Test: ssh sshuser@vps.example.com echo "OK"
    Note: docker-deploy uses your SSH agent or ~/.ssh/config automatically — no flag needed once key is in place.

    H2: "## 2. Passwordless sudo for sshuser"
    Context: docker-deploy may need sudo to create /opt/<project> if it does not exist. Passwordless sudo for specific commands avoids an interactive password prompt during automated deploys.
    Steps:
    1. SSH to VPS as root: ssh root@vps.example.com
    2. Create the deploy user if it doesn't exist: useradd -m -s /bin/bash sshuser && usermod -aG docker sshuser
    3. Edit sudoers safely: visudo -f /etc/sudoers.d/sshuser
    4. Add: sshuser ALL=(ALL) NOPASSWD: /bin/mkdir, /bin/chown, /bin/mv, /bin/rm
    5. Test: ssh sshuser@vps.example.com sudo mkdir -p /opt/test && sudo rm -rf /opt/test
    Note: docker-deploy will prompt for a sudo password interactively if passwordless sudo is not configured. Passwordless sudo is optional but recommended for CI/automated deploys.

    H2: "## 3. Windows users"
    Per D-26: "docker-deploy's install.sh requires bash (or POSIX sh). On Windows, use WSL2 or Git Bash. The binary itself runs natively on Windows via go install if you prefer."

    TROUBLESHOOTING.md content (D-29 — exactly 5 failure scenarios):

    H1: "# Troubleshooting"

    H2: "## 1. SSH authentication failure"
    Symptom: deploy fails with "SSH connection failed: ssh: handshake failed" or "permission denied (publickey)"
    Causes and fixes:
    - SSH key not on VPS: run ssh-copy-id or manually add to authorized_keys (see PREREQUISITES.md)
    - Wrong user: check --host ssh://USER@host — the user must match the key on the VPS
    - SSH agent not running: run eval $(ssh-agent) && ssh-add ~/.ssh/id_ed25519
    - Key passphrase not in agent: ssh-add ~/.ssh/id_ed25519 first
    Verify: ssh sshuser@vps.example.com echo OK — must succeed before docker deploy will work

    H2: "## 2. Unknown host / knownhosts prompt"
    Symptom: deploy stops with a prompt asking you to verify the host fingerprint
    Explanation: docker-deploy uses strict knownhosts verification (no InsecureIgnoreHostKey). On first connection to a new host you must confirm the fingerprint.
    Fix: The prompt shows the server's fingerprint. If you trust the host, type "yes". The host is added to ~/.ssh/known_hosts automatically. Subsequent deploys are non-interactive.
    Alternative: ssh sshuser@vps.example.com echo OK first — this adds the key to known_hosts via standard ssh.

    H2: "## 3. Target directory not writable"
    Symptom: deploy fails with "Deploy failed: mkdir" or "permission denied" on the remote path
    Cause: /opt/<project> does not exist and sshuser cannot create it without sudo.
    Fix:
    - Option A: Configure passwordless sudo as described in PREREQUISITES.md. docker-deploy will create the directory automatically.
    - Option B: Pre-create the directory as root: ssh root@vps.example.com mkdir -p /opt/myapp && chown sshuser:sshuser /opt/myapp
    - Option C: Use --path /home/sshuser/myapp — a user-owned path that doesn't need sudo

    H2: "## 4. Docker not found on remote"
    Symptom: deploy fails with a pre-flight error: "docker: not found" or "docker is not installed"
    Fix: Install Docker on the VPS. Official install: curl -fsSL https://get.docker.com | sh
    Post-install: add user to docker group: usermod -aG docker sshuser; then log out and back in (or run: newgrp docker)
    Verify: ssh sshuser@vps.example.com docker info

    H2: "## 5. docker compose v1 detected (EOL)"
    Symptom: pre-flight fails with "docker-compose v1 detected — docker-deploy requires docker compose v2"
    Cause: The VPS has the legacy docker-compose (v1, Python-based) but not the docker compose plugin (v2, Go-based).
    Fix: Install the Docker Compose plugin: apt-get install docker-compose-plugin (Ubuntu/Debian) or see https://docs.docker.com/compose/install/
    Verify: ssh sshuser@vps.example.com docker compose version — must return "Docker Compose version v2.x.x"
