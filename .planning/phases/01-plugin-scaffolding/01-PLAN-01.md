## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer machine → Docker CLI | The Docker CLI discovers and invokes the plugin binary based on the binary name and location in `~/.docker/cli-plugins/` |
| Build pipeline → binary | GoReleaser cross-compiles binaries; checksums.txt allows download integrity verification |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | `~/.docker/cli-plugins/docker-deploy` install path | accept | Phase 1 delivers a skeleton with no secrets or network access; binary install is developer-controlled. Install requires explicit `make install` — no auto-install. |
| T-01-02 | Tampering | GoReleaser release artifacts | mitigate | `.goreleaser.yaml` (Plan 02) generates `checksums.txt` with SHA256 for each archive. Users verify before installing. |
| T-01-03 | Information Disclosure | `docker-cli-plugin-metadata` JSON response | accept | Only returns plugin name, version string, and description. No secrets, no environment data, no user information. |
| T-01-04 | Elevation of Privilege | `make install` writes to `~/.docker/cli-plugins/` | accept | Install target writes to user home directory only; no sudo, no system paths. Scope limited to current user. |
</threat_model>
