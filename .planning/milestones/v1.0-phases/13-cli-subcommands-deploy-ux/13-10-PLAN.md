<objective>
Fix the `needsSudo` probe incorrectly returning `false` for paths like `/opt/test-deploy` where the user owns the target directory but cannot write to the parent (`/opt`).

The current probe is `test -w /opt/test-deploy || test -w /opt`. When the user owns `/opt/test-deploy`, the first `test -w` exits 0 and the OR short-circuits — `needsSudo=false`. But the `mv /opt/test-deploy /opt/test-deploy-old-XXXXX` operation renames an entry WITHIN `/opt`, which requires write permission on `/opt` (the parent), not on `/opt/test-deploy` itself. This causes the `mv` to fail with exit 1.

The fix: probe ONLY the parent directory. Both first-deploy (`mkdir` creates a new entry in parent) and repeat-deploy (`mv` renames an entry in parent) require the parent to be writable. The target directory's own writability is irrelevant for these operations.

Purpose: Correct the sudo detection logic to match the actual filesystem permission model for `mkdir`/`mv`/`rm` operations.
Output: Single-operand probe in upload.go; updated and extended tests in upload_test.go.
</objective>
