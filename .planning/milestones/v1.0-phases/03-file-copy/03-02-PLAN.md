```go
// filter.go
// ShouldExclude reports whether relPath (relative to the walk root, using
// forward slashes) matches any pattern in excludes.
// Pattern matching uses filepath.Match semantics for each entry.
// Directory entries ending in "/" are matched by checking whether relPath
// starts with the pattern prefix (e.g., ".git/" matches ".git/config").
func ShouldExclude(relPath string, excludes []string) bool

// WalkFiles walks localDir and returns all relative file paths that are NOT
// excluded by ShouldExclude(path, excludes). Directories are never returned.
func WalkFiles(localDir string, excludes []string) ([]string, error)

// upload.go
// Upload copies all non-excluded files from localDir to a staging directory
// on the remote, then atomically renames the staging dir to the final target.
//
// remoteBase is the parent directory (e.g. "/opt/myapp").
// projectName is used to construct the staging dir name: ".deploy-tmp-<timestamp>".
// The staging dir is created under remoteBase's parent:
//   stagingDir = filepath.Dir(remoteBase) + "/" + projectName + ".deploy-tmp-" + timestamp
// The atomic swap: rename remoteBase to remoteBase+".old-<timestamp>", rename
// stagingDir to remoteBase, then remove the old backup. If remoteBase does not
// exist (first deploy), skip the old-backup step.
//
// Progress: prints "Uploading N files..." before starting, then one line per
// file: "  -> relative/path/to/file".
func Upload(ctx context.Context, client *gossh.Client, localDir, remoteBase string, excludes []string) error
```

Note on atomic swap strategy (Claude's Discretion — D-08/D-09 implementation detail):
1. Create staging dir: `remoteParent/projectName + ".deploy-tmp-" + unixTimestamp`
2. Create all directories under staging dir via `sftpClient.MkdirAll`
3. Upload all files into staging dir maintaining relative path structure
4. Via SSH session: if remoteBase exists, `mv remoteBase remoteBase.old-<timestamp>`, then `mv stagingDir remoteBase`, then `rm -rf remoteBase.old-<timestamp>`
5. If remoteBase does not exist (first deploy), just `mv stagingDir remoteBase`

Use `filepath.Dir(remoteBase)` as the parent for the staging dir so staging and target are on the same filesystem (same partition guarantees rename is atomic).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| localDir (filesystem) → SFTP remote | Files read from user's working directory and written to remote over authenticated SSH |
| remoteBase (SSH exec) → mv/rm commands | Remote shell commands constructed from config values |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03 | Tampering | Atomic staging swap | mitigate | Stage to `.deploy-tmp-<timestamp>`, mv to target only after all files land. Partial uploads never reach the live target directory. |
| T-03-04 | Information Disclosure | `.env` file transfer | accept | .env is the core value proposition — it is intentionally copied. No special handling. User is responsible for securing the remote. |
| T-03-05 | Elevation of Privilege | SSH exec `mv`/`rm -rf` commands | mitigate | remoteBase is derived from config.Resolve() which is validated (ParseHost sanitizes the host; path is user-controlled but not shell-expanded). Command strings are constructed with fmt.Sprintf from validated config values, not from raw user input. Staging dir name uses only alphanumerics + timestamp integer. |
| T-03-06 | Denial of Service | Large file upload blocking context cancellation | mitigate | Upload accepts ctx context.Context; pass ctx to session operations where supported. WalkFiles enumerates before uploading so total count is known upfront. |
</threat_model>
